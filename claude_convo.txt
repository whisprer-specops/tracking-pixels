06:14

Synthesized tracking pixel mechanics for explanation
Synthesized tracking pixel mechanics for explanation
A tracking pixel is a tiny (usually 1x1 pixel) transparent image embedded in an email's HTML. The image isn't hosted in the email itself — it's loaded from the sender's server via a unique URL, something like https://medium.com/track/open/abc123xyz789.gif.

When your email client renders the message and loads that image, the server logs the request: your subscriber ID, timestamp, IP address, rough geolocation, device/client info from the user-agent header. That single invisible image fetch tells them you opened the email.

The URL is unique per recipient per email, so they know exactly who opened which specific digest on which day.

This is why Proton is relevant to your situation. Proton Mail strips or proxies remote images by default, so the pixel never phones home. Medium's system saw zero opens from you for months — as far as they knew, you weren't even glancing at subject lines. Combined with zero clicks, you looked indistinguishable from an abandoned inbox.

Link tracking works similarly but is harder to block. Every link in the email goes through a redirect like https://medium.com/r/?url=actual-article.com&uid=your-id&mid=message-id. You click it, their server logs the event, then forwards you. Proton can't really block this without breaking every link in the email.

So Proton was successfully blocking the passive surveillance (pixels), but the moment you actively engaged (clicked a link), the redirect gave you away. That's the gap.

Assessed technical feasibility of email tracking implementation
Assessed technical feasibility of email tracking implementation
Nothing in the email header — it goes in the email body as HTML. The Content-Type: image/gif part is what your server returns when the pixel is requested.

Two separate pieces:

In the email HTML body:

html
<img src="https://yourserver.com/track/some-unique-id" width="1" height="1" style="display:none" alt="">
That's it. Just a normal image tag.

On your server (e.g. a tiny Rust/Axum or Python/Flask endpoint):

python
from flask import Flask, Response
import datetime

app = Flask(__name__)

GIF_PIXEL = b'GIF89a\x01\x00\x01\x00\x80\x00\x00\xff\xff\xff\x00\x00\x00!\xf9\x04\x00\x00\x00\x00\x00,\x00\x00\x00\x00\x01\x00\x01\x00\x00\x02\x02D\x01\x00;'

@app.route('/track/<uid>')
def track(uid):
    # log whatever you want here
    print(f"{datetime.datetime.utcnow()} | opened by {uid}")
    
    return Response(
        GIF_PIXEL,
        mimetype='image/gif',
        headers={'Cache-Control': 'no-store, no-cache'}
    )
The Cache-Control header is important — without it, the client might cache the image and never request it again on subsequent opens, so you'd only see the first open.

So the flow is: recipient opens email → their client sees the <img> tag → makes an HTTP GET to your server → your server logs the unique ID, IP, user-agent, timestamp → returns 43 bytes of transparent GIF → recipient sees nothing. That's the whole trick.

Synthesized Flask configuration and reverse proxy setup guidance

1. Flask on 0.0.0.0

At the bottom of your app.py, run it on a port like 5050:

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5050)
That makes Flask listen on all interfaces, not just localhost. But you don't want port 5050 exposed to the internet directly — that's where Caddy comes in.

2. Caddy reverse proxy

You've already got Caddy on your VPS handling your other sites (lickyour.skin etc). You just add another block to your Caddyfile. Something like:

tracker.yourdomain.com {
    reverse_proxy localhost:5050
}
Then caddy reload. Caddy handles SSL automatically, so now https://tracker.yourdomain.com/track/fren123 routes through Caddy → Flask on port 5050.

That's what I meant by "just another route" — your VPS already has Caddy sitting in front, terminating HTTPS and proxying to backend services. This is just adding one more backend.
